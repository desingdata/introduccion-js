Guía visual y práctica sobre los distintos tipos de errores en JS, cómo manejarlos y patrones para escribir código más robusto.

01 — Tipos de errores nativos
SyntaxError Error de Sintaxis

Ocurre cuando el motor de JS no puede parsear el código por violar las reglas gramaticales del lenguaje. Se detecta antes de ejecutar el script. 

// ❌ Paréntesis sin cerrar → SyntaxError
if (true {
  console.log("Esto no se ejecuta")
}

// ✅ Corrección: cerrar el paréntesis
if (true) {
  console.log("¡Ahora sí funciona!")
}
Error:SyntaxError: Unexpected token '{'
ReferenceError Variable no definida

Se lanza al intentar acceder a una variable que no existe en el ámbito (scope) actual. También aparece al usar variables antes de declararlas con let/const (zona muerta temporal). 

// ❌ Variable inexistente
console.log(precio)  // → ReferenceError

// ❌ Temporal Dead Zone (TDZ)
console.log(x)       // → ReferenceError
let x = 5

// ✅ Declarar antes de usar
let precio = 100
console.log(precio)  // → 100
Error:ReferenceError: precio is not defined
TypeError Tipo incorrecto

Se produce cuando una operación se aplica sobre un valor del tipo equivocado, como llamar algo que no es función, o acceder a propiedades de null/undefined. 

// ❌ Llamar un número como función
const num = 42
num()            // → TypeError: num is not a function

// ❌ Propiedad de null
const obj = null
obj.nombre        // → TypeError: Cannot read properties of null

// ✅ Verificar antes de acceder
const obj2 = null
console.log(obj2?.nombre)  // → undefined (sin error)
Error:TypeError: num is not a function
RangeError Valor fuera de rango

Ocurre cuando se pasa un valor numérico fuera del rango permitido por una función, o cuando se produce un desbordamiento de pila (stack overflow) por recursión infinita. 

// ❌ Tamaño de array inválido
const arr = new Array(-1)   // → RangeError

// ❌ Recursión infinita → stack overflow
function loop() { return loop() }
loop()                         // → RangeError: Maximum call stack

// ✅ toFixed con decimales válidos (0-100)
const pi = 3.14159
console.log(pi.toFixed(2))    // → "3.14"
Error:RangeError: Invalid array length
URIError URI malformada

Se lanza cuando se usan decodeURI() o encodeURI() con caracteres que no forman una secuencia URI válida. 

// ❌ Secuencia de escape inválida
decodeURIComponent("%")      // → URIError

// ✅ URI bien formada
const encoded = encodeURIComponent("hola mundo")
console.log(encoded)           // → "hola%20mundo"

const decoded = decodeURIComponent(encoded)
console.log(decoded)           // → "hola mundo"
OK:hola%20mundo → hola mundo
EvalError Error en eval()

Históricamente relacionado con el uso incorrecto de eval(). Los motores modernos raramente lo lanzan, pero sigue existiendo como clase. Se recomienda evitar eval() por riesgos de seguridad e inyección de código. 

// ⚠️ eval() ejecuta código arbitrario — evitar siempre
const input = "2 + 2"
eval(input)    // → 4  (pero muy peligroso)

// ✅ Alternativa segura para evaluar expresiones
const safeCalc = (a, op, b) => {
  const ops = { '+': (x,y) => x+y, '-': (x,y) => x-y }
  return ops[op]?.(a, b) ?? "Operador inválido"
}
console.log(safeCalc(3, '+', 7))  // → 10
OK:safeCalc(3, '+', 7) → 10
02 — Manejo y captura de errores
try…catch Bloque de captura

Estructura fundamental para capturar errores en tiempo de ejecución. El bloque finally se ejecuta siempre, haya error o no, ideal para liberar recursos. 

// try → código que puede fallar
// catch → captura el error y lo maneja
// finally → siempre se ejecuta (limpieza)
function parsearJSON(texto) {
  try {
    const data = JSON.parse(texto)
    console.log("✅ Éxito:", data)
    return data
  } catch (err) {
    console.error("❌ JSON inválido:", err.message)
    return null
  } finally {
    console.log("🏁 Intento completado")
  }
}

parsearJSON('{"nombre":"JS"}')  // ✅
parsearJSON('texto roto')     // ❌
OK:{ nombre: 'JS' } → 🏁 Intento completado
throw Lanzar errores propios

Permite lanzar errores manualmente con throw. Se pueden crear clases de error personalizadas extendiendo Error, añadiendo propiedades como code o statusCode. 

// Clase de error personalizada con código HTTP
class AppError extends Error {
  constructor(message, code) {
    super(message)         // hereda message y stack
    this.name = "AppError"
    this.code = code
  }
}

function obtenerUsuario(id) {
  if (!id) throw new AppError("ID requerido", 400)
  if (id > 100) throw new AppError("No encontrado", 404)
  return { id, nombre: "Alice" }
}

try {
  obtenerUsuario(200)
} catch (e) {
  console.log(e.code, e.message)  // → 404 "No encontrado"
}
Error:AppError [404]: No encontrado
03 — Errores en código asíncrono
Promise Promesas y .catch()

Las promesas rechazadas se capturan con .catch() o con el segundo parámetro de .then(). Un rechazo no manejado emite el evento unhandledRejection. 

// Promesa que falla si el número es par
const validar =(n) =>
  new Promise((resolve, reject) => {
    if (n % 2 === 0)
      reject(new Error(`${n} es par, se rechaza`))
    else
      resolve(`${n} es impar ✅`)
  })

// .then() para éxito, .catch() para error
validar(4)
  .then(msg => console.log(msg))
  .catch(err => console.error(err.message))
  .finally(() => console.log("Promesa resuelta"))
Error:4 es par, se rechaza → Promesa resuelta
async/await Try-catch asíncrono

Con async/await podemos usar try…catch para manejar promesas rechazadas de forma síncrona y legible, evitando el anidamiento de callbacks. 

// Simula una petición fetch que puede fallar
async function obtenerDatos(url) {
  try {
    // await suspende hasta que la promesa resuelve
    const res = await fetch(url)

    // Lanzamos error si el status HTTP no es 2xx
    if (!res.ok) throw new Error(`HTTP ${res.status}`)

    const json = await res.json()
    return json
  } catch (err) {
    // Captura errores de red y errores HTTP
    console.error("Fallo:", err.message)
    return null
  }
}
Error:Fallo: HTTP 404 → retorna null
Promise.allSettled Múltiples promesas

Promise.all() falla inmediatamente si una promesa se rechaza. Promise.allSettled() espera todas y devuelve el estado de cada una, ideal cuando necesitamos los resultados independientemente de fallos. 

const promesas = [
  Promise.resolve("✅ dato A"),
  Promise.reject(new Error("❌ falló B")),
  Promise.resolve("✅ dato C")
]

// allSettled recoge éxitos Y errores sin interrumpir
const resultados = await Promise.allSettled(promesas)

resultados.forEach(r => {
  // status es "fulfilled" o "rejected"
  if (r.status === "fulfilled")
    console.log(r.value)
  else
    console.error(r.reason.message)
})
✅ dato A | ❌ falló B | ✅ dato C
04 — Patrones y buenas prácticas
Global Handler Captura global de errores

Permite capturar errores no manejados a nivel global en el navegador o Node.js, útil como última línea de defensa para logging y evitar que la app quede en estado indefinido. 

// Navegador: captura errores síncronos no manejados
window.addEventListener("error", (event) => {
  console.error("[Global]", event.message)
  enviarAServidor(event)  // logging remoto
})

// Captura promesas rechazadas sin .catch()
window.addEventListener("unhandledrejection", (e) => {
  console.error("[Promesa]", e.reason)
  e.preventDefault()  // evita log en consola
})

// Node.js equivalente
process.on("uncaughtException", (err) => {
  console.error("[Proceso]", err)
  process.exit(1)  // reiniciar con PM2
})
Log:Error capturado → enviado al servidor de monitoreo
?. ?? Acceso seguro a propiedades

El operador de encadenamiento opcional ?. evita TypeErrors al acceder a propiedades de valores posiblemente nulos. El operador de fusión nula ?? proporciona valores por defecto solo cuando el valor es null o undefined. 

const usuario = {
  perfil: { direccion: null }
}

// ❌ Sin ?. → TypeError si perfil es undefined
usuario.perfil.direccion.ciudad

// ✅ ?. detiene la cadena y devuelve undefined
const ciudad = usuario?.perfil?.direccion?.ciudad
console.log(ciudad)  // → undefined (sin error)

// ?? ofrece valor por defecto solo para null/undefined
const display = ciudad ?? "Ciudad desconocida"
console.log(display)  // → "Ciudad desconocida"

// Llamar métodos que pueden no existir
usuario?.notificar?.()  // seguro aunque no exista
undefined → "Ciudad desconocida" — sin TypeError
Result Pattern Patrón de resultado explícito

En lugar de lanzar excepciones, las funciones retornan un objeto con { ok, data, error }. Hace el flujo de errores explícito y predecible, inspirado en Rust y Go. Evita el uso de try-catch innecesario. 

// Función que retorna Result en vez de lanzar
async function fetchUser(id) {
  try {
    const res = await fetch(`/api/users/${id}`)
    if (!res.ok) return { ok: false, error: "Not found" }
    const data = await res.json()
    return { ok: true, data }       // éxito
  } catch (e) {
    return { ok: false, error: e.message }
  }
}

// Uso: destructurar y verificar ok
const { ok, data, error } = await fetchUser(1)
if (!ok) mostrarError(error)
else    renderizar(data)
{ ok: true, data: { id: 1, nombre: "Alice" } }
